# 入口網站技能執行規則

## 核心原則

不要重新發明入口網站架構。先模仿已成功發布的小伙伴入口網站，再替換成自己的名稱、網址、檔案與技能內容。

## 標準 URL 配置

必須使用同一套路由，避免自己亂命名：

- `index.html`：固定主入口
- `websites.html`：所有網址列表
- `file-manager.html`：Google Drive 檔案總管
- `ppt-catalog/index.html`：互動式 HTML PPT 目錄
- `shadow-skills.html`：影子技能列表

如果是其他小伙伴，把 `shadow-skills.html` 改成該小伙伴代號，例如 `jarvis-skills.html`。

## 執行順序

1. 先抓一個成功案例，確認檔名、相對連結、導覽列和頁尾固定主網頁。
2. 建立相同檔案結構。
3. 替換小伙伴名稱、公開描述、公開網址、Drive 檔案與技能列表。
4. 檢查所有內部連結不能 404。
5. 推上 GitHub。
6. 啟用 GitHub Pages；如果 repo 是 private 且方案不支援 Pages，必須回報阻塞，不要假裝完成。

## 技能說明需要補強的地方

只寫「固定包含主入口、網址列表、HTML PPT 目錄、技能列表、檔案總管」不夠。技能文件必須明確要求：

- 使用標準檔名與相對連結。
- 先模仿成功案例的頁面配置。
- 每個分頁都要有回主入口與其他主功能入口。
- GitHub Pages URL 必須實測。
- private repo 不能公開 Pages 時，要先請 Jason 決定是否轉 public。

## 公開邊界

公開頁只放服務能力、公開作品、公開連結與已授權交付檔案。不放內部 SOP、部署細節、腳本、帳號權限、私人備份或客戶敏感資料。
